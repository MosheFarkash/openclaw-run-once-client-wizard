# Model Routing and QA

Use this for choosing image models, communicating draft/final status to the user, and quality-checking generated creatives.

## Model Strategy

Default strategy:

1. Use **Nano Banana 2 / fast Gemini image model** for cheap, fast draft concepts.
2. Tell the user these are **concept drafts only**.
3. Ask the user to approve the concept direction/style, not the tiny details.
4. After approval, produce final ad-ready images with **GPT Image 2.0** or **Nano Banana Pro**.
5. Run QA before marking anything as ready for ads.

## Preferred Final Models

For final production creatives, prefer:

1. **GPT Image 2.0** — default preferred final model because it is more advanced for polished commercial images, typography-sensitive layouts, and ad-ready creative.
2. **Nano Banana Pro / stronger Gemini image model** — use when Google image generation is preferred, when reference-image editing works better there, or when GPT Image is unavailable.

Use Nano Banana 2 / fast Gemini mostly for:

- First visual explorations.
- Low-cost concept testing.
- Directional mockups.
- Style exploration.
- Showing rough options before spending on final renders.

## User Communication Rules

When showing draft images from Nano Banana 2 / fast Gemini, explicitly say:

```text
These are draft concepts only. Please judge the core direction: hook, layout, style, and idea. Ignore small text mistakes or visual imperfections for now. If you like a direction, I’ll recreate it as a final ad-ready version using GPT Image 2.0 or Nano Banana Pro and run QA before it’s ready for upload.
```

Do not imply draft images are upload-ready.

When generating final images, explicitly say:

```text
Now I’m producing the final ad-ready version using the higher-quality model and then I’ll QA the Hebrew text, layout, and aspect ratio.
```

## Cost Control Rule

Do not spend premium model calls on every early idea unless the user explicitly asks.

Recommended flow:

- Generate multiple draft concepts cheaply.
- Let the user select 1-3 directions.
- Finalize only selected directions with GPT Image 2.0 or Nano Banana Pro.

## Provider Verification Rule

After image generation, check which provider/model actually produced the image.

If the requested provider/model differs from the actual provider/model:

- Record both in the manifest.
- Tell the user if it affects draft/final expectations.
- Do not label an image as GPT Image final if it was actually produced by Gemini draft model.

Manifest fields:

```json
{
  "provider_requested": "",
  "model_requested": "",
  "provider_used": "",
  "model_used": "",
  "generation_tier": "draft | final",
  "is_ad_ready": false
}
```

## Mandatory QA Stage

Run QA after every generated image batch.

Check:

1. Aspect ratio: `1:1`, `9:16`, or `4:3` as requested.
2. Hebrew spelling and grammar.
3. RTL direction and punctuation.
4. Main headline exactness.
5. Supporting text exactness.
6. Text legibility on mobile.
7. Safe margins.
8. Visual clarity and hierarchy.
9. Brand/style consistency.
10. Whether actual provider/model matches requested tier.

## QA Statuses

Use these statuses:

- `draft_concept`: acceptable as rough concept only.
- `needs_edit`: has correct direction but needs regeneration/editing.
- `failed_text`: Hebrew/text errors make it unusable.
- `approved_concept`: user approved direction, ready for final generation.
- `final_candidate`: generated by final model, awaiting QA.
- `ad_ready`: passed QA and can be uploaded.
- `rejected`: should not be used or iterated.

## Hebrew QA Rules

For Hebrew ads, be strict:

- If the main headline has a typo, status is `failed_text`.
- If the CTA has a typo, status is `failed_text`.
- If small decorative text is gibberish but the main text is correct, status can be `needs_edit` or `final_candidate` depending on severity.
- Final ad-ready images must not contain visible gibberish Hebrew/English text.

## Draft Exception

Draft concept images may contain text mistakes if they are clearly communicated as drafts.

For drafts, ask the user to approve only:

- Core concept.
- Composition.
- Visual style.
- Hook idea.
- Direction.

Do not ask the user to approve draft text quality as final.

## QA Output Format

For each image, write:

```markdown
### [file/id]
- Tier: draft/final
- Provider/model requested:
- Provider/model used:
- Aspect ratio: pass/fail
- Text accuracy: pass/fail
- RTL/readability: pass/fail
- Visual hierarchy: pass/fail
- Brand fit: pass/fail
- Status:
- Fix needed:
```

## Finalization Flow

When the user approves a draft concept:

1. Load the draft's prompt lineage.
2. Convert the winning draft direction into a cleaner final prompt.
3. Use GPT Image 2.0 by default, or Nano Banana Pro if selected/available.
4. Preserve the approved composition and hook.
5. Improve text accuracy, polish, safe margins, and ad-readiness.
6. Run QA.
7. Save final output under the same batch or a new `finals` batch.
8. Mark only QA-passed files as `ad_ready`.
